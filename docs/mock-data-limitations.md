# Ограничения mock-данных для демонстрации

> **Версия прототипа:** 0.1.0
> **Дата:** 12 апреля 2026
> **Ветка:** `feat/tz-alignment-v2`

Данный документ описывает все ограничения и заглушки (mock) текущего прототипа. Прототип предназначен **только для демонстрации информационной архитектуры (IA)** и не является production-готовым продуктом.

---

## 1. Хранение данных

### Что сейчас
Все данные хранятся в массивах в `lib/data.ts` и `lib/auth.ts`:
- `MOCK_USERS` — пользователи (4 записи)
- `MOCK_PASSWORDS` — пароли в открытом виде
- `projects` — проекты/маршруты (~15 записей)
- `scenarios` — сценарии конструктора
- `scenarioVersions` — версии сценариев

### Ограничения
| Что | Описание |
|-----|----------|
| **Данные в памяти** | Все изменения (создание пользователей, редактирование, блокировка) происходят в массивах JavaScript. При перезагрузке страницы **данные сохраняются** (т.к. массивы в модуле), но при перезапуске dev-сервера — **сбрасываются** |
| **Пароли в открытом виде** | `MOCK_PASSWORDS` — объект с plain-text паролями. В production требуется хеширование (bcrypt/argon2) |
| **Нет БД** | Отсутствует любая база данных. Все данные — захардкоженные массивы |
| **Нет API** | Отсутствуют REST/GraphQL эндпоинты |

### Демонстрация
- ✅ Создание пользователя работает (данные добавляются в массив)
- ✅ Редактирование пользователя работает (данные обновляются в массиве)
- ✅ Блокировка/разблокировка работает (флаг `isActive` меняется)
- ⚠️ При перезапуске `next dev` все изменения теряются

## 2. Аутентификация

### Что сейчас
- Вход через `contexts/auth-context.tsx`
- Сессия хранится в `localStorage`
- Блокировка после 5 неверных попыток (localStorage, 30 минут)
- Проверка `roleExpiryDate` при входе

### Ограничения
| Что | Описание |
|-----|----------|
| **Нет серверной сессии** | Сессия полностью на клиенте (localStorage). Можно подделать |
| **Rate limiting на клиенте** | Блокировка через localStorage. Можно сбросить очисткой хранилища |
| **Нет JWT** | Токены не генерируются, авторизация через JS-объект |
| **Нет CSRF/XSS защиты** | Отсутствуют заголовки безопасности, cookie-based сессии |
| **Пустой пароль = вход** | `login('ivanov', '')` работает всегда (тестовый режим) |

### Демонстрация
- ✅ Вход с паролем работает
- ✅ Блокировка после 5 попыток работает
- ✅ Первый вход со сменой пароля работает
- ✅ Восстановление пароля (запрос администратору) работает
- ✅ Истёкшие роли блокируют вход

## 10. Известные проблемы

| Проблема | Влияние |
|----------|---------|
| Данные сбрасываются при перезапуске dev-сервера | Нужно перезаходить после рестарта |
| Нет обработки ошибок сети | Прототип не делает сетевых запросов |
| `MOCK_USERS` мутируется напрямую | При concurrent-модификациях возможны коллизии |
| Нет валидации email на уникальность | Можно создать двух пользователей с одинаковым email |
| Нет проверки сложности пароля | Временный пароль — случайная строка, но без требований |

## 11. Рекомендации для демонстрации

### Сценарий демо (15-20 минут)
1. **Вход как `ivanov`** (пустой пароль) → показать KPI-дашборд
2. **Перейти в Проекты** → показать список, сравнение, ранжирование
3. **Перейти в Конструктор** → показать сценарии, сравнение
4. **Выйти, войти как `admin`** → показать администрирование
5. **Создать пользователя** → показать генерацию пароля, toast
6. **Карточка пользователя** → показать блокировку/разблокировку
7. **Аналитика → Отчёты** → показать формирование экспорта (toast)
8. **Профиль** → показать редактирование, toast

### Что говорить заказчику
- «Это прототип информационной архитектуры»
- «Все данные — заглушки для демонстрации навигации»
- «Бэкенд будет подключён на следующем этапе»
- «Экспорт файлов будет работать после интеграции с бэкендом»
